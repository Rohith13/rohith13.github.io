Storefront Options — When to Use What

B2C Commerce offers four distinct storefront approaches. Recommending the right one is a core SA competency.

SFRA — Storefront Reference Architecture

  • Server-rendered MVC (ISML templates)
  • Cartridge-based, runs on B2C Commerce app server
  • Mature, widely deployed, large SI ecosystem
  • Best for: lift-and-shift, fast go-live, budget-constrained

Composable Storefront (PWA Kit)

  • React + Node.js SPA on Managed Runtime (MRT)
  • Headless, uses SCAPI exclusively
  • Full frontend flexibility, slower initial build
  • Best for: brands needing custom UX, omnichannel

Storefront Next GA 26.6

  • Next.js-based, server-side rendered, on MRT
  • Opinionated quick setup, Page Designer native
  • Best server-side security model (no client tokens)
  • Best for: new builds wanting speed + modern stack

BYOS — Bring Your Own Storefront

  • Any tech stack (Vue, Angular, custom React)
  • Uses SCAPI for commerce functions
  • No MRT — customer hosts their own frontend
  • Best for: enterprise with existing tech investment

Cartridge Model

SFRA is built on a cartridge pipeline. Cartridges are modular code packages that stack on top of each other.

Cartridge Path Resolution

B2C Commerce resolves controllers, templates, and scripts by walking the cartridge path left-to-right. First match wins.

Controller Override

To customize, place a controller with the same name in your custom cartridge and call server.extend(base) or server.append/prepend/replace to modify behavior.

ISML Templates

Server-rendered HTML templates. Use <isinclude>, <isloop>, <isif> tags. Extend templates by overriding in your cartridge path.

Script API (dw.*)

Server-side JavaScript API to access B2C data: dw.catalog.ProductMgr, dw.order.OrderMgr, dw.customer.CustomerMgr, etc.

SFRA Analytics Tagging

SFRA: Strengths & Limitations

Strengths

  • Largest SI ecosystem and partner knowledge base
  • Mature accelerators for B2B, luxury, marketplace patterns
  • No additional hosting cost (runs on B2C app server)
  • Built-in Page Designer support (ISML components)
  • Session-based cart + checkout is simpler than headless
  • Proven at scale — thousands of live sites

Limitations

  • No native PWA/offline capability
  • Server-rendering means slower perceived performance
  • Cartridge conflicts in complex multi-vendor setups
  • Frontend flexibility is constrained (ISML, not React)
  • Mobile Core Vitals often need extra optimization
  • B2C Commerce investment path is now composable

Key Technology Stack

React + Express

React for UI components; Express (Node.js) handles SSR routes on the MRT App Server. Isomorphic — same code runs server and client side.

Retail React App (RRA)

The reference implementation. Provides PDP, PLP, Cart, Checkout, Account pages out of the box. Built with Chakra UI for accessible components.

commerce-sdk-react

React hooks library for SCAPI. Replaces the old CommerceAPI class. Uses TanStack Query for data fetching and caching. Hooks: useProduct, useBasket, useSearchResults, etc.

Template Extensibility

Override specific files from the base RRA template in your overrides/ directory. Reduces code footprint — you only own what you customize.

PWA Kit Request Flow (SSR)

Browser requests page

Request hits MRT's CDN edge. If cached → returns immediately. If not → routes to App Server.

Express App Server receives request

Node.js + Express handles routing. React components render on the server (SSR) — full HTML returned to browser for fast First Contentful Paint.

SCAPI calls made server-side

App Server calls B2C Commerce SCAPI using a SLAS token (private client — token never leaves server). Product, search, basket data fetched.

HTML hydrated on client

React "hydrates" the server-rendered HTML — attaches event handlers. Subsequent navigation is SPA (client-side routing, no full page reload).

TanStack Query caches data client-side

commerce-sdk-react uses TanStack Query — API responses cached in memory. Stale-while-revalidate pattern keeps UI fast.

MRT Proxy & Caching

Proxy (Uncached)

Path prefix: /mobify/proxy/. Routes requests to configured origin. Bypasses CDN — origin always called. Use for personalized or session-sensitive API calls.

Caching Proxy (Cached)

Path prefix: /mobify/caching/. CDN caches response. Strips Cookie from cached requests. Use for public catalog data, product images, static content.

B2C Commerce APIs

Do NOT use MRT's caching proxy for SCAPI calls. B2C Commerce has its own server-side web-tier cache — use that instead.

CDN Cache Vary Headers

MRT's CDN creates separate cache entries for each unique combination of these headers:

Accept, Accept-Charset, Accept-Encoding, Accept-Language, Authorization, Range

Multi-MRT Architecture — The Anti-Pattern

Zlatin's #1 Architecture Correction Each MRT environment has its own isolated CDN cache pool. More environments = fragmented caches = lower cache hit rates = more origin calls = higher latency. A customer with 4 MRT environments (2 brands × 2 regions) should consolidate to 1–2 MRTs and use multi-site routing in config/sites.js.

Base Paths (PWA Kit v3.17.1+)

The envBasePath property in config/default.js sets a base path for all Express routes. This was the feature that resolved locale routing issues in multi-brand setups, reducing the need for separate MRT environments per brand/region.

    url: { showBasePath: true }

Key Storefront Next Features

Server Components (Next.js 13+)

React Server Components run exclusively on the server — zero JS bundle shipped for server-rendered content. Faster page loads, better Core Web Vitals.

Native Page Designer Integration

Out-of-the-box support for Page Designer headless components. No manual ComponentRegistry setup. Merchants get visual editing on the live Next.js storefront immediately.

Opinionated Setup

Comes with sensible defaults for routing, API configuration, SLAS setup, and Page Designer. Fewer architectural decisions means faster time-to-launch compared to PWA Kit's blank-slate flexibility.

Custom site/global preferences fetched during SSR and injected via window.__MRT_DATA_STORE__. No separate API call from browser to get preferences.

SLAS tokens stored in HttpOnly cookies — inaccessible to client-side JavaScript. Eliminates the primary XSS attack vector for token theft.

Mass Product Deletion — The 900k Products Problem

Real scenario from American Red Cross (Zlatin's case). Customer wants to delete 900,000 obsolete products.

Option 1: SCAPI Delete Product (DON'T)

900,000 individual DELETE calls. No batch. Not practical — rate limits, duration, operational overhead.

Option 2: Import/Export Job API (BEST)

Job API reintroduced in SCAPI 26.5. Use import job with delete mode. Bulk, async, platform-optimized.

Option 3: Minimal XML (QUICK WIN)

Not as scary as it sounds. Only needs catalog wrapper + product ID + delete flag. No full product data required.

Option 4: Custom API via Script API

Write a custom endpoint using dw.catalog.ProductMgr.removeProduct() in a loop. Full control, runs on-platform.

<!-- Minimal XML for bulk product deletion -->
<?xml version="1.0" encoding="UTF-8"?>
<catalog xmlns="http://www.demandware.com/xml/impex/catalog/2006-10-31"
         catalog-id="storefront-catalog">
    <product product-id="PROD-001">
        <mode>delete</mode>
    </product>
    <product product-id="PROD-002">
        <mode>delete</mode>
    </product>
    <!-- Repeat for all 900k products — generate programmatically -->
</catalog>
No Full Product Data Needed for Deletion Customers assume XML deletion requires full product XML. It doesn't. Only the catalog wrapper, product ID, and delete mode are needed. This is often the quickest path to bulk deletion while SCAPI's Job API matures.

OCAPI → SCAPI Migration Strategy

Audit existing OCAPI usage

Map every OCAPI endpoint in use. Categorize: Shop API (shopper-facing) vs Data API (admin). Identify what has a SCAPI equivalent.

Run OCAPI and SCAPI in parallel

No flag-day switch. New shopper-facing features use SCAPI. Legacy OCAPI integrations continue untouched.

Migrate shopper APIs first

SCAPI Shopper APIs are the most mature. Migrate product search, product detail, cart progressively.

Admin/data APIs last

Job API in SCAPI 26.5+ covers bulk operations. Custom APIs bridge remaining gaps. OCAPI Data API can stay longer.

What is SLAS?

SLAS is Salesforce's authentication layer for B2C Commerce Shopper APIs. Every SCAPI call requires a SLAS token. Each SLAS client is registered to a single tenant (one B2C Commerce instance).

Client Types — The Fundamental Decision

The One Question to Ask Can your application securely store a client secret? If YES → Private Client. If NO → Public Client (PKCE).

Public Client (PWA Kit default)

  • No client secret stored
  • Uses PKCE for security
  • Access token in browser memory/storage
  • Single-use refresh tokens
  • Use for: SPAs, mobile apps, any client-side app
  • Risk: tokens accessible to JS, XSS vulnerable

Private Client (Storefront Next / BFF)

  • Has client secret (stored server-side)
  • Uses client_credentials grant for guests
  • Token never leaves server — stored in HttpOnly cookie
  • Reusable refresh tokens
  • Use for: full-stack apps, BFF, Storefront Next
  • Best security: no client-side token exposure

PKCE Flow — Explained Simply

PKCE: How Security Works Without a Client Secret
  1. Client generates:
     code_verifier  = "random_43_to_128_char_string"
     code_challenge = BASE64URL(SHA256(code_verifier))

  2. Client → SLAS /authorize?
     code_challenge=abc123...
     code_challenge_method=S256
     ← SLAS stores challenge, returns auth_code

  3. Client → SLAS /token?
     code=auth_code
     code_verifier=random_43_to_128_char_string    ← Original string
     ← SLAS verifies: SHA256(verifier) == stored challenge
     ← Issues: access_token + refresh_token

  Why secure: Attacker intercepting auth_code cannot redeem it
              without the code_verifier (never transmitted in step 2)

Public vs Private Refresh Token Behavior

Public Client

Refresh tokens are single-use. Each time you use a refresh token, SLAS issues a new one. Old token is invalidated immediately.

Private Client

Refresh tokens are reusable. The same token is returned in responses. Allows simpler server-side session management.

The Multiple Client ID Anti-Pattern (Nintendo Case)

Critical: Multiple SLAS Client IDs Cannot Share Refresh Tokens SLAS disabled cross-client refresh token reuse. A user authenticated under Client ID A cannot use their refresh token under Client ID B. This breaks the user journey if they navigate between areas using different client IDs.

  User navigates to account → Client ID B expects its own token
  Client ID B rejects Client ID A's refresh token → FORCED RE-AUTH → bad UX

  ✅ Correct: Single Client ID per storefront

  User authenticates once → Single Client ID issues token + refresh
  Token works across all pages: browse, cart, checkout, account, order history
  Security isolation achieved through SCOPE design, not separate clients

When Multiple Client IDs ARE Valid

  • Genuinely separate storefronts with different user bases (B2B portal + B2C site)
  • Different instances/tenants with no shared sessions
  • NOT for: different functional areas within the same user journey
  • NOT for: "security isolation" between checkout and account — use scopes instead

Scope Hygiene — Common Mistake

Don't Mix Shopper and Admin Scopes Shopper scopes belong ONLY on SLAS clients. Admin/data scopes belong ONLY on Account Manager clients. Mixing them causes "token bloat" and unexpected behavior — the scopes don't work as expected when combined.

Token Revocation Edge Cases

  • When a shopper changes password → SLAS revokes previously issued access tokens
  • Revocation takes several minutes — there's a window where old tokens remain valid
  • Token revocation is only default-enabled in production — non-production may not revoke
  • Rate limit: repeated calls to same endpoint with same USID within short window → HTTP 409

Search Performance Anti-Patterns

Too Many Searchable Attributes

Every searchable attribute increases index size and build time. Audit periodically — remove unused searchable flags.

Thousands of Variants per Master

Each variation product is an indexable entity. 50k+ variants per master causes index bloat. Use Variation Groups to aggregate.

SKU-Level Merchandising

Merchandising individual SKUs (variation products) rather than masters or VGs creates search noise and operational complexity.

Unscheduled Index Rebuilds in Production

Triggering index rebuilds during peak traffic causes performance degradation. Schedule off-peak, monitor duration.

Boost & Bury GA 26.6

Platform-native search result merchandising. Merchandisers can boost specific products to the top of results or bury them to the bottom — based on search terms, categories, or product attributes. This is the preferred path forward vs. custom search ranking code in SFRA cartridges.

Performance Optimization & Troubleshooting

A core SA delivery — architecture health reviews with measurable performance outcomes (60% response time improvement on Product-Show is a real team example).

SFRA Performance Checklist

Page Cache Coverage

Verify page cache is enabled and correctly configured for Home, Category, Search, and PDP. Check cache hit rate in Log Center. Low hit rate = high origin load.

Script API Efficiency

Audit dw.* API calls in controllers. N+1 query patterns are common: fetching product availability one-by-one in a loop instead of batch. Use ProductMgr.getProduct() vs looping ProductSearchHit.

Searchable Attribute Count

Too many searchable attributes increases index build time and search response time. Audit and remove unused searchable flags. Target: only attributes that shoppers actively filter on.

Custom Objects & Queries

Inefficient Custom Object queries are a top SFRA performance culprit. Always query with an indexed attribute. Avoid full scans. Set appropriate cache TTL on CO data.

3rd Party Script Bloat

Tag managers, chat widgets, analytics scripts loaded synchronously block rendering. Defer non-critical JS. Use async/defer attributes. Enforce a tag governance policy.

Image Optimization

B2C DIS supports WebP, lazy loading parameters. Ensure images are requested at correct viewport size — don't serve 2000px images for 400px thumbnails.

PWA Kit / Composable Storefront Performance

SSR Cache Hit Rate

Monitor MRT CDN cache hit rate. Low = MRT App Server handling too many requests. Fix: review Cache-Control headers on SSR responses, ensure public pages don't set session cookies.

Bundle Size

Use analyze-build script (webpack-bundle-analyzer) to identify large dependencies. Code-split at the route level — don't ship PDP code on the homepage.

TanStack Query Stale Time

Set appropriate staleTime for commerce-sdk-react hooks. Product data doesn't change every second — a 5-minute stale time prevents unnecessary SCAPI refetches.

SCAPI Response Time

Profile individual SCAPI endpoint response times. Shopper Search is often the bottleneck — check search index build recency, searchable attribute count, refinement config complexity.

Performance Diagnostic Flow

Baseline with real metrics

Get Time to First Byte (TTFB), Largest Contentful Paint (LCP), Core Web Vitals. Use Log Center (MRT) or Business Manager log files (SFRA). Don't guess — measure first.

Check CDN/cache hit rate

High TTFB → likely a cache miss problem. Check page cache configuration, verify cache-busting isn't happening on every request (e.g., session cookie being set on cacheable pages).

Profile server-side execution

SFRA: check controller execution time in logs. PWA Kit: Lambda execution duration in Log Center. Identify slow operations: API calls, Script API queries, custom logic.

Check data layer efficiency

Search index build recency (stale index = slow search). Custom Object query patterns. Replication queue depth (long queues = stale data + overhead). Catalog complexity (attribute count, variation count).

Quantify and document improvement

SA engagements are outcome-focused. Measure before and after. The team example: 60% improvement in Product-Show response times. Always attach a number to your recommendation.

Search Performance Specifically

  • Run a search index rebuild and measure duration — baseline for catalog health
  • Check Search Processor log for slow queries: anything >200ms is worth investigating
  • Reduce searchable attributes to only what's needed — most impactful single change
  • Check locale-specific index configuration — a misconfigured locale index affects all searches in that locale
  • Review boost/bury rules complexity (26.6+) — overly complex rules can add latency

Architectural Scenarios

End-to-end architecture decision exercises. These are the scenarios SAs actually navigate — know how to reason through them.

Scenario 1: Net-New B2C Commerce Customer — Which Storefront?

Setup: Mid-market fashion retailer. 50 SKUs, single region (US), merchant team of 5. No existing tech. 6-month go-live target. Limited SI budget.

Recommendation: Storefront Next or SFRA

  • Storefront Next if the team wants a modern stack with Page Designer from day 1 — opinionated defaults, fast setup
  • SFRA if they need maximum SI partner availability and accelerator reuse — large partner pool, proven at this scale
  • NOT PWA Kit — too much React setup overhead for a small team with a hard deadline
  • NOT BYOS — no existing tech investment to protect

Scenario 2: Multi-Brand Global Retailer — MRT Architecture

Setup: Two brands (Garage, Dynamite), two regions (US, Canada). Currently on SFRA. Migrating to headless. Team has 4 MRT environments planned.

Recommendation: 1–2 MRT environments

  • Single MRT with multi-site routing in config/sites.js — best cache hit rate
  • Or 1 MRT per brand (2 total) if brands have truly separate deployment cadences
  • NOT 4 MRTs — CDN cache fragmentation, lower hit rates, higher latency, higher operational overhead
  • Verify PWA Kit 3.17.1+ — base paths feature resolves locale routing without extra MRTs
  • Shared basket works across brands — doesn't require separate MRTs

Scenario 3: SLAS Security Audit — Financial Services Retailer

Setup: Customer wants maximum security for SLAS tokens. Currently using public client (PKCE). Considering multiple Client IDs for checkout vs browse.

Recommendation: Private client + Storefront Next (server-side)

  • Switch to private client — client secret stored server-side, token never reaches browser
  • Enable HttpOnly session cookies (PWA Kit 3.18+ / Storefront Next) — XSS cannot steal tokens
  • NOT multiple Client IDs — refresh tokens cannot be shared across client IDs, breaks user journey
  • If greenfield: Storefront Next is the right answer — server-side architecture eliminates client-side token exposure entirely
  • Add strict client auth header (x-slas-client-auth) to prevent credential stuffing

Scenario 4: 900k Product Catalog Cleanup — No Full XML

Setup: American Red Cross. 900,000 obsolete products. Customer insists on SCAPI, refuses XML.

Recommendation: SCAPI Job API (26.5+), then minimal XML as fallback

  • First option: SCAPI Import/Export Job API — reintroduced in 26.5, designed for bulk operations
  • Second option: Custom API using Script API ProductMgr.removeProduct() in batch
  • Third option: Minimal XML — only catalog wrapper + product ID + delete mode. No full product data needed.
  • NOT 900k individual SCAPI DELETE calls — rate limits, impractical duration
  • File VOC for native SCAPI batch delete capability

Scenario 5: Customer Accuses Platform of Wrong Search Results — Red Account

Setup: Vishal Mega Mart. Search relevance issues. Customer blaming B2C Commerce platform. Executive escalation.

Recommendation: Two-session structure — buy-in first, deep-dive second

Session 1 — Get customer buy-in

Demonstrate — live, with the customer's own data — that their search configuration choices are creating the relevance problems. Don't be defensive. Show evidence. Shift from adversarial to collaborative.

Session 2 — Deep dive and recommendations

Full walkthrough of how search works, how to troubleshoot it, specific configuration fixes, catalog data quality recommendations. Close the red account with documented next steps.

Scenario 6: Large-Scale Customer Segmentation — 4 Million Customers

Setup: Natura & Co. Want to target 4 million specific customers with a promotion. Considering static customer groups.

Recommendation: Data Cloud activation to segments array

  • Build segment in Data Cloud → activate to B2C Commerce → writes segment ID to customer.segments array
  • Create Dynamic Customer Group: rule = customer.segments CONTAINS [segment-id]
  • Assign promotion to that Dynamic Customer Group
  • NOT static customer groups — quota limits, import overhead, not real-time
  • NOT custom attributes — long-term governance burden, query efficiency degradation
  • For guest shoppers: single-use coupons distributed via Marketing Cloud
  • Governance: stop running 100s of replications per day in Production

Scenario 7: SFRA → Composable Migration Strategy

Setup: Enterprise retailer, heavily customized SFRA, 200+ active customizations. Wants to migrate to headless.

Recommendation: Incremental hybrid migration

Audit customization inventory

Categorize all 200+ customizations: pure frontend (easy to migrate), business logic (needs rewrite), integrations (needs SCAPI equivalent). This determines the actual migration cost.

Hybrid storefront — PWA Kit pages layered on SFRA

B2C Commerce supports hybrid: SFRA serves most pages, PWA Kit takes over specific high-value pages (e.g., PDP first). Session Bridge handles auth handoff between the two.

Migrate page by page

Each page migrated = one less SFRA cartridge to maintain. Measure performance improvement per page to build business case for full migration.

Evaluate Storefront Next for remaining pages

If Storefront Next is GA and stable, evaluate it for pages not yet migrated — faster setup than PWA Kit, better Page Designer support for content-heavy pages.

Zlatin Questions — Technical Depth

A customer has 4 MRT environments for 2 brands × 2 regions. What's your recommendation and why?
I'd recommend consolidating to 1–2 MRT environments. MRT doesn't map 1:1 to B2C Commerce sites — a single MRT can serve all four combinations using locale path routing in config/sites.js. The problem with 4 environments is CDN cache fragmentation: each has its own isolated cache pool, so you get much lower hit rates than a consolidated setup. MRT Engineering confirms fewer environments equals better cache utilization. I'd check they're on PWA Kit 3.17.1+ since the base paths feature resolves the locale routing issues that often drive customers to separate MRTs. At most I'd suggest 2 MRTs — one per brand — if there are genuinely separate deployment cadences. The shared basket concern doesn't force separate MRTs — that's at the SLAS/B2C Commerce layer, not MRT.
Why would you recommend Storefront Next over PWA Kit for a new implementation?
Three main reasons. First, security: Storefront Next uses a server-side private client architecture — SLAS tokens never touch the browser. PWA Kit public client puts the access token in browser memory, exposed to XSS. Second, setup speed: Storefront Next is opinionated with sensible defaults — faster time-to-launch because you're not making architectural decisions from scratch. Third, Page Designer: Storefront Next ships with native Page Designer support out-of-the-box, while PWA Kit requires manual setup of ComponentRegistry and SLAS scope configuration. The tradeoff is that Storefront Next is less flexible than PWA Kit for unconventional UX — so for very custom storefronts, PWA Kit is still the right tool.
What's the difference between OCAPI BatchRequest and what SCAPI offers for bulk operations?
OCAPI BatchRequest is a proprietary Salesforce feature that bundles multiple operations into a single HTTP call — useful for high-volume admin operations. SCAPI intentionally didn't port this because SCAPI is designed as a standards-based REST API, and proprietary batch patterns don't fit that model. For bulk operations in SCAPI, the options are: the Job API reintroduced in SCAPI 26.5 for import/export operations, Custom APIs using Script API commands for mass operations, or XML import for catalog operations — where many customers are surprised to learn that deletion only requires the catalog wrapper and product ID tags, not full product data. I'd also file a VOC for a native SCAPI mass-delete capability since it's a genuine product gap.
Walk me through the PKCE flow and why it's secure without a client secret.
PKCE works by replacing the client secret with a dynamically-generated challenge/verifier pair. The client generates a random code verifier, then creates a code challenge by hashing it with SHA-256. In the authorization request, the client sends only the challenge — not the verifier. The auth server stores the challenge. Later, when the client exchanges the auth code for a token, it sends the original verifier. The auth server verifies that SHA-256(verifier) matches the stored challenge. Security comes from the fact that an attacker who intercepts the auth code cannot redeem it without the original verifier — and the verifier was never transmitted in the vulnerable initial request. This is why it's safe for browser-based apps where a static client secret would be trivially extractable from the JavaScript bundle.

Erika Questions — Precision Platform Knowledge

A customer says their CCAC New customer count doesn't match their JDBC analysis. 46% of orders show as New. What's happening?
Two likely causes, both undocumented. First, DNT orders: when shoppers have Do Not Track enabled, the platform sets buyer_email to the literal string "___ANONYMOUS___". CCAC sets purchase_count to 0 for these and always classifies them as New — it's platform-intended but never documented. If 46% of their shoppers have DNT enabled or are guest sessions with no email captured, that directly explains the discrepancy. Second, realm-level scoping: CCAC looks up purchase history across ALL sites in the realm, not just the current site. If this is a multi-brand tenant and the customer's JDBC query only looks at one site's orders, they'll always get a different New count than CCAC. I'd help them adjust their JDBC query to match CCAC's logic, and I'd file a VOC to separate genuine New from anonymous/DNT New, plus a documentation request for the realm scoping behavior.
A customer's PDP shows no selectable color or size options after a catalog import completed without errors. Diagnose it.
This is almost certainly unpopulated variation attributes. B2C Commerce allows catalog XML to import successfully even when variation products have their variation attribute values set to "-None-" or left blank — the platform is designed to allow staged imports where you might add attribute values later. The result is a master product with variation products that have no actual color or size values assigned. In Business Manager, navigate to the product's Variations tab — you'll see variation products with attribute values showing "-None-". To prevent this in future imports, you can mark the variation attributes as mandatory at the attribute definition level — this catches unpopulated values at import time. But that's not universal, since not all products share the same attributes. The real fix is upstream: the catalog creation process needs to ensure attributes are populated before generating the import file. Existing broken products need to be re-imported with correct values.
A customer asks for training on their custom Page Designer implementation. How do you scope this SA engagement?
This is a scope misalignment I'd flag immediately. SA engagements are advisory — we cover platform best practices, architecture guidance, and feature education on standard Salesforce capabilities. Training a customer's team on a custom Page Designer implementation that was built by their SI partner is out of scope for Success Architects. That type of work falls to their implementation partner. I'd set this expectation clearly with the CSM before the engagement, then offer a scoped alternative: a standard Page Designer advisory session covering platform capabilities, best practices, headless component architecture, and common anti-patterns. If they need broader coaching on their custom implementation, the Success Guide route or a paid SI engagement is the right path.
A customer wants to target 4 million customers in B2C Commerce using static customer groups. Why is that a problem and what do you recommend?
Static customer groups are designed for distributed data — they work well for small, relatively stable lists. At 4 million members in a single group, you'll hit quota limits and performance problems. Every time the audience changes, you need another full import cycle. There's no real-time update capability — the audience membership always lags behind the source data. The right path is Data Cloud segment activation. Build the segment in Data Cloud, activate it to B2C Commerce. Data Cloud writes a segment ID to the segments array on each customer's profile object. In B2C Commerce, create a Dynamic Customer Group with a rule checking customer.segments CONTAINS that segment ID, and assign your promotions to it. This has been validated at 10 million activations with no performance concern. It's real-time, no custom attributes, no import overhead. For guest shoppers who can't be looked up by profile, single-use coupons distributed via Marketing Cloud are the clean alternative.

Scenario-Based Questions

A new B2C Commerce customer is deciding between SFRA, PWA Kit, and Storefront Next. What questions do you ask before recommending?
Six qualifying questions: (1) Timeline — how many months to go-live? Under 6 months pushes toward SFRA or Storefront Next. (2) Team skills — do they have React/Node developers? If not, SFRA is safer. (3) SI partner — which partner is implementing? Do they have PWA Kit expertise or just SFRA? (4) Content management needs — how dependent are merchants on Page Designer? Storefront Next has the best out-of-box Page Designer support. (5) UX complexity — standard commerce UX or highly custom? Custom pushes toward PWA Kit for maximum flexibility. (6) Budget — MRT hosting adds cost; SFRA runs on the B2C app server at no additional charge. Then I'd map the answers to the decision framework: fast go-live + limited budget → SFRA; modern stack + fast setup + Page Designer → Storefront Next; maximum UX flexibility + strong React team → PWA Kit; existing tech investment → BYOS.
